<p><strong>Introduction</strong></p> <p>One of the tough things about DDDD right now is there isn’t a mountain of information available like with other programming topics.&# 160; I guess that’s a blessing in disguise because there’s also not a ton of <strong>mis</strong>information either.</p> <p>In traditional N-tier systems and in repository-based, DDD systems, we typically let the database manage all of our state for us.&# 160; Then we use different patterns, such as the <a href="http://martinfowler.com/eaaCatalog/optimisticOfflineLock.html">optimistic offline lock</a> (AKA optimistic concurrency), to ensure concurrency when multiple processes are trying to modify the same information at roughly the same time.</p> <p>In DDDD things work a little bit differently.&# 160; Once you start to scale a bounded context out beyond one server, you have to worry about concurrency—meaning that we ensure that the understanding of the known world is consistent between any instance of the same bounded context.&# 160; This is a very challenging problem, but it is one that has been solved in <a href="http://www.amazon.com/Pattern-Oriented-Software-Architecture-Distributed-Computing/dp/0470059028">several different ways</a>.&# 160; What we need to decide is how best to solve the problem in a distributed, message-based environment.</p> <p><strong>The Identity Map</strong></p> <p>As I have <a href="/2009/03/infinite-loop-event-sourcing.html">blogged about previously</a>, Greg mentions that each bounded context has an identity map and that we can use pub/sub to notify all other instances of the same bounded context of changes in state as a result of processing commands.&# 160; This works great, but once we have multiple instances running, we need to be extremely careful lest two instances of the same aggregate process commands which produce conflicting results.</p> <p><strong>An Example</strong></p> <p>Let’s suppose there’s a user on eBay who wants to sign-up with a certain username—let’s say it’s “bob” and, for the sake of argument, that it’s available.&# 160; Because of the number of users using eBay, it’s a fairly probable situation that he could attempt to register with that username at the same time as someone else.</p> <p>If the same aggregate was on multiple servers and a request came into both instances simultaneously requesting the username “bob”—one from <a href="http://www.objectmentor.com/omTeam/martin_r.html">Bob Martin</a> and the other from Bob Jones—all of a sudden, we have a problem.&# 160; Each aggregate has approved the username “bob” and passed a message back to some workflow saying that it’s reserved the name for their respective user.&# 160; We’ve got a concurrency issue.</p> <p>If not address properly we will run into some very nasty temporal bugs that may or may not be discovered for a long time.&# 160; What’s worse, the longer the bug exists, the more damage it causes to all related systems.&# 160; Usually these types of bugs have a way of corrupting data not just in their own bounded context, but throughout all parts of the extended domain.</p> <p><strong>The Solution</strong></p> <p>Rather than randomly having any instance of the same bounded context receive commands that it needs to perform, we instead want to forward them to a single instance.&# 160; You might say, “Wait a second!&# 160; Why run multiple instances if all commands are forwarded to only one instance?!”&# 160; The answer is simple: Partitioning.&# 160; We can use a locator (devTeach @ 46:20) to forward the command to the appropriate instance of the bounded context based upon some selection criteria.&# 160; In the case of our eBay example—we could partition based upon the first letter of the username, such that all requests for usernames starting with the letter “b” are forwarded to a specific instance of the bounded context for the aggregate root to process.</p> <p>Continuing with our example—the same aggregate would now service both requests.&# 160; It would service one command and then the other.&# 160; The first would be approved, the second would be denied.&# 160; Usernames starting with other letters, would be handled by other instances.&# 160; Because we have different instances handling non-overlapping portions of work, we have effectively eliminated concurrency issues—for this particular context.&# 160; Do you see how we can distribute and scale?</p> <p>The only caveat, per my understanding, is that we need to ensure that each aggregate only performs one unit of work at a time—effectively <strong>serializing all work</strong>.&# 160; Serializing (queuing up work), makes it seem like commands would back up very quickly.&# 160; Fortunately because of ways that we can <a href="/2009/03/event-sourcing-and-committing.html">architecture our DDDD infrastructure</a>, each unit of work does not take long to complete—especially when we don’t have to communicate with a persistence engine directly and virtually all communication is in-process communication.</p>
