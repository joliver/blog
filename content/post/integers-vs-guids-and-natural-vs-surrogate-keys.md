<p>In the database world, there are few arguments that provoke such strong feelings as the proper choice for a database key.&# 160; Should we choose a natural or surrogate key?&# 160; If we choose a surrogate key, should it be an integer or a GUID?&# 160; I have actually done work on all sides of the argument, so I can see it from the various perspectives.</p> <p><strong>Natural vs. Surrogate</strong></p> <p>Joe Celko argues that we should be using a natural key--one composed from the attributes of the entity being modeled rather than just assigning an arbitrary value.&# 160; I agree with him, but don't stop reading yet.&# 160; We should keep our keys thin (small) rather than creating a massive composite key, whose values would also be necessary in other tables in order to perform a join.&# 160; How do I reconcile these two arguments?&# 160; Easy, I do both.</p> <p>Allow me to elaborate.&# 160; I actually have the database's primary key be the natural key, but I create a meaningless, arbitrary surrogate key that can be used for joins in related tables, among other things.&# 160; In this way I get good performance on joins as well as an easy way to select rows, but I enforce the natural key of the thing being modeled to ensure that two aren't accidentally inserted at the same time.</p> <p><strong>Integers vs GUIDs</strong></p> <p>To start, can we please say it &quot;<a href="http://en.wikipedia.org/wiki/Globally_Unique_Identifier">gwid</a>&quot; rather than &quot;goo-id&quot;?&# 160; This aspect of the argument is also heated, but the irony is that the surrogate key camp is fighting amongst itself.</p> <p>I have used both extensively, but recently there has emerged a clear winner: the GUID.</p> <p>The main argument between ints vs GUIDs stems from space usage--4 bytes vs. 16 bytes--yes, GUID are 16 bytes, not 36.&# 160; The argument is that the database is faster when working with 4 byte keys.&# 160; True, the integer it can be marginally faster, but it has a number of significant disadvantages.&# 160; (Update: The other compelling argument is related to index fragmentation and page splits.&# 160; This is easily solved by using a GUID “COMB”—see the comments for more information.)</p> <p>First of all, databases are smart.&# 160; When doing key comparison, they don't have to compare all 16 bytes in order to determine inequality among keys, they stop comparing at the first bit that differs.&# 160; Furthermore, when performing a select+join, rarely do I want to pull an object with thousands or millions of rows associated with it--when the size difference between integers and GUIDs would start to become measurable.&# 160; In most applications, when you perform a select, it's only retrieving a dozen rows or so--perhaps maybe a few hundred, but much beyond that you're properly returning too much for the user to digest and all of the associated object creation would become the bottleneck instead of the key comparison.</p> <p>Second, in any mid-size to large system, e.g. eBay, PayPal, etc. you're going to need an 8-byte number, e.g. BigInt, otherwise you'll exhaust all available values in the database.&# 160; Ever had to quickly update your database schema and application code to handle 8-byte long values?&# 160; A local company, Authorize.Net, famous for gateway processing, had this problem because they're used signed, 4-byte integers.&# 160; They had to send out a notice to all there customers of the change.&# 160; Fortunately I treat all their return values and identifiers values as strings, so we weren't affected.&# 160; But your application won't ever get this big right?&# 160; I hope all of my projects become wildly successful and I plan accordingly.</p> <p>Third, and this is the biggest one for me, in a 100% fully distributed system we want to avoid a single point of contention by creating a &quot;shared nothing&quot; architecture--including the database.&# 160;&# 160; By having a GUID rather than an integer, we can have the application code generate the object's persistence identifier and then persist to the database without first talking to the database.&# 160; This removes additional burden from the database, reduces latency, improves application performance, allows for occasionally connected systems, and removes the database as a shared resource in a distributed system.&# 160; Note that using MySQL Cluster you can then geo-distribute your system (but you’ll still pay for latency).&# 160; In a geo-distributed system you definitely don't want to have auto-increment integers because you're just asking for problems...big, big problems.</p> <p>Fourth, a GUID adds a small measure of security by obscurity.&# 160; About two years ago we ordered a few Dell machines for our office.&# 160; When I got to the checkout's &quot;thank you&quot; page, I noticed the parameter &quot;&orderId;=12345678&quot; in the query string.&# 160; Just for kicks I subtracted one from the number to yield 12345677 and then I pressed enter.&# 160; All of a sudden on my screen I could see someone else's order information--their name, email, shipping and billing address as well what they ordered.&# 160; I did this a few more times and sure enough, each time I got a lot of customer information.&# 160; I thought to myself, <em>I could write a script to harvest all of the order information</em>.&# 160; Then I thought, <em>nah!</em> and continued with my day.&# 160; This was Dell we're talking about!&# 160; They didn't have proper authorization checks in their code.&# 160; One small advantage to GUIDs is the <em>security by obscurity</em> value.&# 160; Of course you want to code your application properly, but it's just a small, additional layer of protection that it gives you.</p> <p>Fifth, GUIDs can help ensure that you have written your database joins (via NHibernate) properly.&# 160; Back when I used to hand-code stored procedures, I would get a join wrong and get some weird results that may or may not have been quickly discovered.&# 160; By using GUIDs your joins won't work unless you code them properly.</p> <p><strong>Conclusion</strong></p> <p>The arguments for on both sides are often compelling.&# 160; We have simply found that the use of GUID makes life a lot easier and allows us to focus on other things rather than something as silly, yet mission critical as a primary key.</p> <p>-------------------</p> <p>UPDATE: While I am still a firm believer in GUIDs, it’s interesting to see what Twitter does for their IDs:</p> <p><a href="http://www.royans.net/arch/distributed-systems-and-unique-ids-snowflake/">http://www.royans.net/arch/distributed-systems-and-unique-ids-snowflake/</a></p> <p><a href="https://github.com/twitter/snowflake">https://github.com/twitter/snowflake</a></p>
