<p>In a <a href="/2009/07/serializing-queuing-messages-to.html">previous post</a>, I detailed all of the reasons why processing messages one by one with an aggregate root was a good thing.&# 160; I am now going to contradict some of the main points in that post.&# 160; While certain aspects still hold true, there is a much larger matter involved.</p> <p>First, the points that I still consider to be valid:</p> <ol> <li>Don’t have threading logic in your aggregate roots.&# 160; Don’t do things with locks, monitors, mutexes, etc.&# 160; Your domain objects should run on a single thread at a time.&# 160; They’re going to be complex enough as is with business concerns, why add so much incidental complexity related to threading and infrastructure to cloud their intent? </li> <li>Don’t use synchronous calls to services within your aggregates.&# 160; Send a message to the service.&# 160; The service will, at some point in the future, publish a message asynchronously back to the aggregate and the aggregate will continue processing. </li> </ol> <p>While messaging patterns easily allow us to distribute the various/different bounded contexts (subsystems) across multiple machines, how do we handle splitting a single bounded context?&# 160; <a href="/2009/03/dddd-locatornot-single-point-of-failure.html">My initial understanding</a> was to us some kind of partitioning strategy with <a href="http://tech.groups.yahoo.com/group/domaindrivendesign/message/14088">advanced routing mechanisms</a>.&# 160; This type of strategy may be overkill for most scenarios.</p> <p>The simple answer is optimistic concurrency—sort of.&# 160; Optimistic concurrency tells us that, while we were working with a copy of an object, the original object has changed underneath us.&# 160; Greg has mentioned <a href="http://tech.groups.yahoo.com/group/domaindrivendesign/message/12615">IConflictWith</a> as a way to solve this issue.&# 160; But there’s still something missing.&# 160; IConflictWith is only designed to deal with *commands* that you send to your domain.&# 160; When a domain object receives two commands that conflict, it may ultimately reject one or both.&# 160; Events are another matter.&# 160; Events happened and must be dealt with—they cannot be rejected.</p> <p>In a single machine/context environment, things are easy.&# 160; In fact, it’s the simplest thing that could possibly work.&# 160; The problems that you may ultimately encounter in this scenario relate to fault tolerance/single point of failure and vertical scaling to name a few.&# 160; For bounded contexts in your domain that require a timely response due to strict SLAs, you absolutely must have some kind of fault tolerance.&# 160; This requires distributing a single bounded context such that it runs on multiple machines simultaneously.</p> <p>The big problem running a single bounded context on multiple machines relates to consistency within aggregates.&# 160; If an aggregate can run on machine A and machine B simultaneously, who’s right?&# 160; Who is “consistent”?&# 160; They both are.&# 160; By using messages, we send all appropriate events to the other instance of the bounded context so that it can be made aware of what happened in our instance.&# 160; But that introduces yet another problem still: what happens if they can’t communicate?</p> <p>If you have two machines, you can virtually guarantee that the network between them will be&# 160; unavailable at some point in time.&# 160; But if the network is unavailable, how do we ensure that the events between copies of the same aggregate are handled by the other copies quickly?&# 160; How can each copy be made aware of the events from the other copies?&# 160; They can’t.&# 160; At least, not until network connectivity is restored.</p> <p>Suppose you have geo-distributed your application.&# 160; One critical bounded context of the application is running on a server in Los Angeles and another copy of that same bounded context in New York.&# 160; If for some reason that network between them goes down, each piece will continue to operate and service requests.&# 160; If the network is down for a few minutes, hours, possibly even days, each piece can continue to process and be available.</p> <p>In the above scenario, what are the chances that each piece may have performed some work that was also performed by the other?&# 160; What are the chances that each copy has started to diverge from the other?&# 160; The chances are pretty good—especially as more time passes.&# 160; How do we handle this?&# 160; How do we reconcile conflicting differences between two copies of the <strong>same</strong> aggregate object?&# 160; Compensating actions.</p> <p>At some point in the messaging pipeline—perhaps as late as the reporting bounded context—something will detect the anomaly showing that these two aggregate instances have produced conflicting results.&# 160; Whoever detects the anomaly sends a message back into the aggregate letting it know of the conflicted state.&# 160; The aggregate would then take the appropriate compensating action in order to reconcile the differences.</p>
