<p>For the past few weeks I have been scouring the Internet investigating all things related to <a href="http://codebetter.com/blogs/gregyoung/archive/tags/DDDD/default.aspx">DDDD</a> as well as the concepts behind <a href="http://codebetter.com/blogs/gregyoung/archive/2008/04/28/dddd-10-cqs.aspx">Command Query Separation</a> at an architectural level.</p> <p>One of the main posts that is referenced is Martin Fowler's work on <a href="http://martinfowler.com/eaaDev/EventSourcing.html">Event Sourcing</a>.&# 160; In a nutshell, the article talks about capturing each change to the application state as an object and then persisting those objects.&# 160; Then, to rebuild the state of an application at a given time, you simply re-run those captured objects against your entity.</p> <p>It's a really powerful concept but there were several caveats that I wanted to iron out in my head.&# 160; After watching Greg's <a href="http://codebetter.com/blogs/gregyoung/archive/2008/05/20/devteach-talk.aspx">devTeach talk</a> I was finally able to figure it out.</p> <p>My problem was that, as I read the Martin's article, it sounded like he was capturing the commands coming into the object rather the state on the other side as a result of the commands.&# 160; Perhaps it was the example he used that gave me this impression.&# 160; From that limited perspective, Event Sourcing seemed much more difficult.&# 160; Specifically, if I was re-running all of the commands against an object at a later point, but the code had changed, such as a feature change or a bug fix, wouldn't my final state be vastly different when the object was reconstituted?&# 160; The short answer is yes.</p> <p>As I watched the devTeach video, I realized that the messages that Greg was capturing were actually the event messages coming out of the entity which had explicit state transitions in them.&# 160; These weren't the commands coming into the entity.</p> <p>This is is a fundamental and critical difference.&# 160; If I were to reprocess a command against the entity, I very well may have different state coming out as the underlying code changes.&# 160; But when reprocessing events, the story is much, much different.&# 160; In many ways, reprocessing an event message is like hydrating your entity from a database.&# 160; The entity just looks at the message which says, &quot;The ABC variable was XYZ at this point in time&quot;, and it sets the variable accordingly.</p> <p>Granted, there are a few small versioning issues related to changes in your event messages, but messaging as a whole has much better guidance on how to version, e.g. with constructive rather than destructive changes.</p>
