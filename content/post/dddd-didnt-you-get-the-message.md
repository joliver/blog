<p>Greg has generously shed some light on a critical aspect of DDDD: How do I get the command (or event) message into the aggregate root?</p> <p>First a small amount of background.&# 160; When a message arrives for consumption, typically a message handler like MapMessageToAggregateRootHandler will be responsible for loading the aggregate from the repository and pushing the message into the aggregate root.&# 160; (Note: The messages handled by this handler all have a GUID property representing the aggregate ID which the handler can use to load the aggregate from the repository.)&# 160; From a high level, the concept is easy enough.&# 160; But what about implementation specifics?&# 160; What does the code actually look like to get the message into the aggregate root?</p> <p>Originally I was thinking that the aggregate root would somehow need to expose a method for every single type of message (command or event) that it could consume.&# 160; This could work, but it would make the message handler bloated and tightly coupled to the aggregate root.</p> <p>Then I thought about having a single HandleMessage(IMessage msg) method on an aggregate root.&# 160; This is better (in some respects) because at least the application service is protected from the internals of the domain.&# 160; But still it’s a big smelly method!&# 160; A single HandleMessage method with a massive switch statement is just the kind of procedural code we’re trying to get away from.</p> <p><strong>The Elegant and Simple Solution</strong></p> <p>Greg informed me that his aggregates explicitly implement an interface of:</p> <blockquote> <p>IConsume<t> where T : Message</t></p> </blockquote> <p>Here’s the beauty of the solution.&# 160; Your application service would load the aggregate from the repository.&# 160; It would then cast the aggregate to IConsume<t> and it would then push the message into the aggregate through the Consume() method of the IConsume interface.&# 160; Brilliant!</t></p> <p>The reason this works so well is:</p> <ol> <li>The message handler is completely decoupled from domain-specific methods. </li> <li>The aggregate root doesn’t need a single point of entry for all messages.&# 160; Each message comes through its own point of entry. </li> </ol>
