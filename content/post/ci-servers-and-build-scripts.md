<p>About a year ago, I read <a href="http://www.amazon.com/Continuous-Integration-Improving-Software-Reducing/dp/0321336380">Continuous Integration: Improving Software Quality and Reducing Risk</a>. It’s a great book that talks about the importance of making your CI server the hub of development.</p> <p>One part of the book that was very insightful was the exact role of build scripts.&# 160; Specifically, the build scripts should be able to execute <strong>on our developer machines as well as the CI server</strong>.&# 160; Even though we were attempting to do this to some extent, there were some additional advantages that were obtained by going the rest of the way.</p> <p><strong>Choose Your Server</strong></p> <p>When we first started doing automated builds, we wrote scripts under CruiseControl which ran exclusively on the server.&# 160; There was no possible way to run the scripts on the developer machine short of having a full-blown CC instance running on our dev machines.&# 160; The XML scripts were error prone and bugging and tightly coupled us to a CruiseControl-specific implementation.&# 160; Further, everything was lumped together into one monolithic build script which made us afraid to touch it for fear of breaking the build.</p> <p>All CI products out there—CruiseControl, Cruise, TeamCity, Hudson, Continuum, etc. provide the ability to invoke activities, but I am not aware that they provide guidance stating that you should have CI-independent build scripts which are simply invoked through the CI server.</p> <p><strong>Separation of Concerns</strong></p> <p>To break our build server dependency, we moved away from CI-specific scripts and chose a “portable” solution—at first NAnt and finally rake.&# 160; We then focused on writing build scripts that were broken down and focused on several different types of activities:</p> <ol> <li>Code compilation</li> <li>Static analysis/best practice compliance</li> <li>Testing in its various forms</li> <li>Source control actions (e.g. tagging)</li> </ol> <p>By having smaller script activities, we could compose them into larger once as necessary.&# 160; Further, it allowed us to narrowly focus our attention on a single concern rather than mixing code compilation with source control actions, for example.</p> <p>In this way, we became CI server independent—we could pick whichever CI server implementation that we wanted because it would simply call a series of scripts.&# 160; This also allowed us to have a private developer build to be performed locally prior to each commit (or “git push”).&# 160; The local developer would be virtually assured that his source code changes would integrate because the developer build script was identical to the one that would be executed on the CI server.</p> <p><strong>Separate Workspaces</strong></p> <p>A best practice for build servers is to break down longer running tasks into shorter tasks.&# 160; Usually you’ll want to have some sort of fast-running “integration” task that compiles and does a few unit tests.&# 160; This is a minimum threshold of quality that code must meet to be considered acceptable code.&# 160; Beyond that, any number of additional long-running scripts may execute and perform a full range of testing against the code.&# 160; Finally, at the end, some kind of “publish” is done to make the build artifacts generally available at a well-known location.</p> <p>One big problem that we noticed was that in a local or developer environment, your workspace is “stable” in that you can run a particular build script and immediately be ready to run the next build script because the binaries are at a specific location.&# 160; A CI server provides no such luxury.</p> <p>A CI server almost always has a separate workspace (directory) for each “build project” that will be executed against a development project (i.e. a single C# project or solution).&# 160; This presents a problem because the “Integrate” project has no way to push the compiled artifacts into the “Acceptance Testing”, “Performance Testing”, and “Code Metrics” projects.</p> <p>Here we had to help our CI server a little bit.&# 160; We simply modified our “Acceptance Testing” project to retrieve the build artifacts (using “wget”) from the “Integration” project prior to kicking off the acceptance-testing related scripts.</p> <p>Other than that, our CI server simply calls out to our build scripts.&# 160; In this way, we maintain the ability to “hot swap” our build process at a moments notice.</p> <p>And by the way, we’re using Hudson.</p>
