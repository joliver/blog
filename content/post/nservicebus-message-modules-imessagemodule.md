<p align="justify">When handling messages in NServiceBus it may become necessary to consume the same message multiple times but in different ways.&# 160; For example, suppose you wanted to update separate yet related tables in the database.&# 160; You could throw all of the business logic into a single message handler and be done with it.&# 160; The only problem is that this message handler would be performing multiple actions and the intention behind the handler would be obscured.&# 160; Instead, create multiple, distinct message handlers and have each perform one complete action related to the message, e.g. updating a table and publishing a message.&# 160; In this way it becomes much easier to reason about each handler independently.</p> <p align="justify"><strong>Logical vs. Physical Messages</strong></p> <p align="justify">When NServiceBus receives a message from a message queue, it receives what is known as a TransportMessage.&# 160; The TransportMessage can be viewed as a message envelope of sorts.&# 160; It holds multiple business/application messages and gives some metadata about its return address, correlation identifier, deliverability guarantees, and time to be received.</p> <p align="justify">NServiceBus receives a TransportMessage from the queue and deserializes it, typically using the default XmlSerializer.&# 160; Deserializing this physical message yields a collection of logical business/application messages.&# 160; Each logical message is then dispatched to the list of registered handlers.</p> <p align="justify">But what if we wanted to wrap all of the handlers that we’re executing?&# 160; What if we wanted to have some behavior occur prior to the handlers receiving messages and after all handler code was complete?&# 160; We need message modules.</p> <p align="justify"><strong>Using Message Modules</strong></p> <p align="justify">At the most basic level a message module is a .NET class that implements the IMessageModule interface which is found in the NServiceBus.dll assembly.&# 160; This interface exposes three parameterless methods: HandleBeginMessage(), HandleEndMessage(), and HandleError().&# 160; These methods are invoked exactly once per physical TransportMessage received.&# 160; In a typical scenario the only methods invoked will be HandleBeginMessage() and HandleEndMessage().</p> <p align="justify">Message modules are designed to wrap the receipt of a physical message.&# 160; They allow you to set values and invoke behavior during the initial stages of receiving a message and to perform any cleanup after all handlers have executed.&# 160; Lastly, they allow you to invoke additional behavior in failure scenarios.</p> <p align="justify">The canonical example of implementing a message module is when working with NHibernate’s ISession and ISessionFactory.&# 160; During HandleBeginMessage(), we create a new session and bind that session to the thread, e.g.</p> <blockquote> <p align="justify">CurrentSessionContext.Bind(sessionFactory.OpenSession());</p> </blockquote> <p align="justify">This allows each message handler executing on the same thread to obtain the same instance of ISession and thus participate in the same unit of work when interacting with a database.&# 160; When all logical messages have been handled NServiceBus will invoke the module’s HandleEndMessage() method where we can unbind the current session from the thread and perform additional clean as necessary.</p> <p align="justify"><strong>Message Module Behavior</strong></p> <p align="justify">There are a few things to be aware of when implementing a message module.&# 160; The first is that message modules are global or singleton scoped.&# 160; When the bus is started it requests the set of configured message modules from the configured IoC container.&# 160; This set of modules is then held and stored for the duration of the bus lifetime.</p> <p align="justify">Because a message module is singleton scoped, care must be taken not to store message-specific state directly in the module.&# 160; Instead, this information should be stored on the thread performing the processing by decorating the associated static member variable with the TheadStatic attribute.&# 160; In addition NServiceBus reuses threads.&# 160; This means that all values stored on the thread using the ThreadStatic attribute must be cleared and reinitialized at the start of processing a new message.&# 160; Failure to reset all state within a module may lead to subtle bugs in your infrastructure code.</p> <p align="justify">Multiple message modules can be registered to execute.&# 160; Simply have each implement IMessageModule.&# 160; NServiceBus will automatically scan all assemblies in the runtime directory and register any classes that implement IMessageModule.&# 160; Generally, the ordering of message modules should be unimportant to your application code.&# 160; Nonetheless, you may explicitly control the ordering of the message modules by registering each with your IoC container explicitly and in the order that best fits your requirements.&# 160; Message modules are always invoked from first to last.&# 160; This means that when HandleBeginMessage() is called, it is called on the first registered module then on the second registered module and so on.&# 160; This same ordering—first to last—is maintained when invoking HandleEndMessage() and HandleError().</p> <p align="justify">When a message module is registered with the bus, at least two of the three methods will always be invoked—HandleBeginMessage() and HandleEndMessage().&# 160; In error or failure conditions these two methods will be invoked and then HandleError() will be invoked.&# 160; This means that there isn’t a “Finally()” or “Dispose()” method for each module.&# 160; That is, there’s no way to be sure that HandleEndMessage() is the last method to be invoked because it may be followed by HandleError() in failure conditions.</p> <p align="justify"><strong>Conclusion</strong></p> <p align="justify">Despite a few minor quirks in the current implementation message modules serve a valuable purpose inside of an application and can be leveraged when the situation dictates their use, just like any other tool in the expansive NServiceBus toolbelt.</p>
