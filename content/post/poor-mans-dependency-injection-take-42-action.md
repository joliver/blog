<p>I watched the battle from the sidelines where Jimmy Bogard talked about how not to do “<a href="http://www.lostechies.com/blogs/jimmy_bogard/archive/2009/07/03/how-not-to-do-dependency-injection-in-nerddinner.aspx">poor man’s dependency injection</a>”.  Then Tim Barcz gave <a href="http://devlicio.us/blogs/tim_barcz/archive/2009/07/12/why-there-s-nothing-wrong-with-dependency-injection-in-nerddinner.aspx">his input</a>.  Then Chad Myers got <a href="http://www.lostechies.com/blogs/chad_myers/archive/2009/07/14/just-say-no-to-poor-man-s-dependency-injection.aspx">into the mix</a>. Considering this blog is called Inversion of Control Freak, I should probably add my $.02 now that things have settled down.</p> <p>I’ve gone back and forth on this issue where I really want to allow flexible wire-up of my components using the IoC container of choice, but at the same time, I’d like to have the ability to just skip the wire-up and have a default, ready-to-go implementation.   In other words, I want all the power and flexibility and all the convenience at the same time.  I want my cake and I want to eat it too!</p> <p>Rinat Abdullin, co-creator of the coolest IoC container <a href="http://code.google.com/p/autofac/">Autofac</a>, had a good idea that I had kicked around in his <a href="http://abdullin.com/journal/2009/7/24/couchdb-in-the-cloud-persisting-from-net-code.html">most recent blog post</a>.  Previous to reading his post, I had considered just creating a simple, default factory class that would do the default wire-ups and go from there.  Rinat, in his article, said the following (about 2/3 the way down the page) regarding the creation of a factory <em>method</em>:</p> <blockquote> <p>By the way, that's a good approach for creating developer-friendly libraries. We implement all functionality as separated and decoupled components (as in component-driven development) ready to be consumed by whatever IoC Container the developer likes. Extensibility and flexibility come in there natively. </p> <p>Should the project be too simple for wring up the proper IoC infrastructure, developer-friendly library will have some static factory [method] to use. This class would serve as a logical entry-point into the library and would host hard-coded creation of components (using recommended configurations and settings).</p> </blockquote> <p>There’s the answer.  Just create a simple factory method.  Those that want to swap default implementations can do so using constructor-based injection, while those that just want to use the library without all of the fuss can do so as well.</p>
