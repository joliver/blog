<p>I’ve been working with CQRS for almost a year.&# 160; Ever since I first heard about it, I was immediately convinced that it had the ability to vastly simplify a lot of unnecessary development overhead.&# 160; In this particular post, I will not be discussing event sourcing which is often mingled with CQRS.&# 160; Instead, I will be focusing on the query side of the equation.</p> <p>Please note that the ideas contained herein are merely conjectures of what might be rather than any kind of working, concrete implementation.&# 160; I’ve thought my way through some aspects and ramifications of implementing what I mention below but new insights will undoubtedly merit additional consideration.&# 160; Furthermore the solutions outlined below need not be considered mutually exclusive with more traditional ways of doing things.&# 160; There is definitely room to implement portions of what I mention for specific views while doing things in the traditional way in other views.</p> <p><strong>ViewModel Tables</strong></p> <p>Several weeks ago <a href="/2009/11/cqrs-reporting-database-access.html">I blogged</a> about having one table per view.&# 160; Since that time Udi Dahan <a href="http://www.udidahan.com/2009/12/09/clarified-cqrs/">has mentioned</a> it as well, although independently of my post.&# 160; Querying view model tables directly, which comes as a positive consequence of implementing CQRS, enables us to avoid having numerous mapping layers between the database and the view.&# 160; Further, it allows us to avoid projecting off of our domain model and keeps the domain model focused solely on behavior.&# 160; Lastly, it allows us to scale the read database independently of the write database.</p> <p><strong>Content Delivery Networks</strong></p> <p>Think for a moment about the pure static resources that are handled on a given website.&# 160; They just sit there with the exact same content for each request.&# 160; These are the exact kinds of resources for which we could leverage a content-delivery network.&# 160; In other words, we could push these static elements to a CDN and have the CDN deliver them for us thus offloading&# 160; traffic from our servers and freeing our web tier to perform more interesting tasks—such as processing update-type commands.&# 160; Not only does this offload work from our web tier, but the user has a better and faster browsing experience.</p> <p><strong>HTML + JSON</strong></p> <p>In ASP.NET as well as other server-based page-rendering technologies we typically have the request come all the way to our server and then we render some HTML but with a few modifications depending upon the user.&# 160; Could there be a way to somehow render static layout HTML but render the parts that vary separately?&# 160; Why not have the pages render identical static HTML regardless of user and then, on a separate request, render the user-specific content and then stitch the two together using JavaScript?</p> <p>Let me say that another way.&# 160; Our MVC view or our ASP.NET “Page” could render pure HTML with absolutely no server-side markup.&# 160; Inside of that HTML we would have the URL of an embedded resource: <script src=""></script>.&# 160; The request to this embedded resource would call our MVC controller and then render JSON data with user-specific information for the currently logged-in user for that view.&# 160; We could then use a client-side JavaScript templating engine such as jTemplates to combine the JSON with the static HTML.</p> <p><strong>Static HTML Views + CDNs</strong></p> <p>The above has some interesting ramifications.&# 160; Why couldn’t we just put the static HTML for each view into an HTML file and push those HTML files to a CDN?&# 160; The static HTML files would have <script> tags inside which would make HTTP requests to our controller actions.&# 160; Of course we would need to figure out a way to render each view with its associated header/footer HTML or perhaps use JavaScript to get the various HTML parts of the page and assemble them on the client.</p>  <p>By doing this, our web tier is now responsible for exactly two things: outputting user-specific data in JSON format and handling of mutating/update-type POST commands from the browser.</p>  <p><strong>CDNs + JSON</strong></p>  <p>Here’s where things get interesting.&# 160; What if we didn’t have to query the database through our controller/service layer to render the JSON?&# 160; What if the request for the user-specific data never reached the web server?&# 160; What if the user-specific JSON files were <em>already</em> on the CDN?&# 160; Think about that for a minute.&# 160; If that was somehow possible, our web servers would only be responsible for handling POSTs which contained task-specific commands.&# 160; Wouldn’t that make the web tier massively scalable?&# 160; Wouldn’t it be easier to handle exponentially larger loads on the web tier without the need for additional hardware?&# 160; Even if we didn’t use a CDN and we did everything ourselves, Apache and IIS are really, really good at serving up static files when compared to the expense of going through the ASP.NET pipeline.&# 160; [This applies to any server-side technology such as Ruby, PHP, Perl, Java, etc.]</p>  <p>How is this accomplished?&# 160; In addition to what has been mentioned throughout this post of splitting up the HTML and the JSON requests, there are two additional pieces.&# 160; First, you would want to have a component subscribing to events being published by your domain model.&# 160; This component would then render the user-specific JSON for each view or page.&# 160;&# 160; Second, we would need a way to map the incoming request for a particular JSON file at a “general” location, e.g. “/scripts/account-history” to a user-specific location, e.g. “/scripts/user-12345/account-history.js”.</p>  <p><strong>URL Rewriting + Session State ID + Hard Links</strong></p>  <p>That last part is the tricky part because we want to ensure that the user’s information is protected and cannot be accessed by others.&# 160; The way to do it is with URL rewriting and file system hard links/junction points.&# 160; The user-specific JSON file would be written to a known/fixed location, but that location couldn’t be accessed and would never be disclosed over the web.&# 160; Instead what we would do is rewrite the URL to be: “/scripts/<session-state-id>/account-history.js”.&# 160; This could be accomplished through either server-side rewriting or even client-side rewriting.</p>  <p>If you recall, the session state ID is very, very temporary.&# 160; When a user logs out of their account or the session expires, the session state is nullified and disappears.&# 160; But we don’t typically think like that about directories.&# 160; This is where hard links or junction points come into play.</p>  <p>When a user logs in, a hard link is created which points to the user-specific (but never disclosed) directory.&# 160; For example “/scripts/<session-state-id>/account-history.js” would map to: “/scripts/<random-guid>/account-history.js”.&# 160; When the login takes place, a small, server-side lookup is performed to get the GUID for the particular user—not the user identifier, but another GUID that is never publicly disclosed.&# 160; We then have a server-side process create a hard link between the temporary session state identifier and the GUID.&# 160; Conversely, when the user logs out or the session expires, the hard link is removed.&# 160; In this way we can protect the user data from prying eyes—there would be no way to access the user data without knowing the session state identifier.</p>  <p><strong>Conclusion</strong></p>  <p>All of this is highly experimental, but conceptually makes sense.&# 160; The development tools currently available bring us some but not all of the way there.&# 160; Further, there is an amount of overhead in developing such a solution, but the benefits could be significant, especially the separation of concerns aspect and the ability to scale without significant additional effort.&# 160; Amazon, eBay, Google, and other big players currently implement some or even most of the above techniques but I don’t think I’ve ever seen all of them put together in quite this way.</p>  <p>There are three other benefits that I just thought of.&# 160; First, if your web servers are completely offline, because of HTTP content caching with expirations, your users wouldn’t even notice—until they tried to perform an update/POST.&# 160; In this way, your website remains highly available—even if your servers aren’t online!&# 160; The second benefit, although much smaller, is that you could more easily render browser-specific implementations of your site, such as those for mobile devices and browsers (which you can already do this using different views in MVC).&# 160; Lastly, Google can index your site much more quickly because everything is effectively static, which will soon have a positive effect on your PageRank.</p>  <p>For those of you that have read thus far, I’m curious to hear your comments.</p>  </script></p>
