<p>In my <a href="/2009/03/concurrency-in-dddd-world.html">previous blog post</a>, I elaborated upon Greg’s <a href="http://codebetter.com/blogs/gregyoung/archive/2008/05/20/devteach-talk.aspx">devTeach talk</a> in which he hints at distributing a bounded context.&# 160; I talked about how using what Greg calls a “locator” can facilitate distribution without introducing concurrency problems.</p> <p>The question was asked in his talk at 46:40, “Is the locator a single point of failure?”&# 160; Because his talk was not actually about distributed systems, but rather about CQS and some foundational aspects of DDDD, Greg pulled back from that question and moved on.</p> <p>Here’s the answer:&# 160; No, it’s not a single point of failure.</p> <p>If you had a single instance of a “locator”, yes, it would be.&# 160; But if you had multiple instances, then you’d be okay.&# 160; Great, now we reintroduced a whole mess of concurrency issues by having multiple locators.</p> <p><strong>Enterprise Integration Patterns</strong></p> <p>Before we dig into solving concurrency between each instance of the locator, we should probably use the appropriate name for the solution.&# 160; In Gregor Hohpe’s book, <em>Enterprise Integration Patterns</em>, he talks about a pattern he calls a <a href="http://www.eaipatterns.com/ContentBasedRouter.html">Content-based Router</a>.&# 160; The “locator”, per Greg’s talk, seems to be content-based router.&# 160; The word “locator” to me sounds much more request/response-oriented, while a content-based router sounds much more like its Ethernet counterpart, which forwards messages (or packets) to an endpoint.&# 160; Therefore, rather than calling it a “locator”, I will simply refer to it as a content-based router or “router” for short.</p> <p><strong>Semi-stateful Routing</strong></p> <p>The content-based router could either be implemented with hard-coded rules (stateless), or it could function in a much more dynamic fashion (stateful).</p> <p>In Greg’s devTeach talk, he talked about how the router would be aware of CPU time and available memory on each node to which it could forward a command.&# 160; In his <a href="http://www.infoq.com/interviews/Architecture-Eric-Evans-Interviews-Greg-Young">interview with Eric Evans</a>, he talked about ensuring that no two ticker symbols in his stock trading system would live on the exact same node—thus drastically reducing the probability that multiple node failures would bring down any ticker symbol.</p> <p>From a performance standpoint, the router makes all of its decisions based upon information available in memory, rather than using any kind of external service call or database.&# 160; In this manner, the router would perform as if it was completely stateless, while in fact, it remembers on which node a particular aggregate root lives.</p> <p><strong>Single Point of Failure</strong></p> <p>The router has the responsibility of determining the node to which commands will be sent and remembering that information.&# 160; By itself, it’s a single point of failure.&# 160; To avoid this, we simply apply the same pub/sub model to the router.&# 160; Whenever a decision is made regarding routing tables, we publish that information to other instances of the router.&# 160; In this manner, we avoid the router being a single point of failure.</p> <p><strong>Concurrency Revisited</strong></p> <p>If we have multiple routers, don’t we have to manage concurrency between them?&# 160; Yes, if they process messages simultaneously.&# 160; But if we only have one instance of the router active at a time and listening for messages, then we don’t have a problem.</p> <p>How do we ensure that one and only one router is listening for messages to forward?</p> <p><strong>Stayin’ Alive / Another One Bites The Dust</strong></p> <p>At this point, we should introduce an infrastructure message, the heartbeat message.&# 160; At a specified interval each instance of a bounded context AND each router would publish a heartbeat message and publish it to each router saying “I’m alive.”&# 160; We can use this message and embed a few other tidbits of information—including available memory or CPU time, etc.</p> <p>This message serves two purposes.&# 160; First, if a particular node misses X hearbeats, we know that the node has failed and we can re-direct the messages to another node.&# 160; We then notify all other routers of the same so that they know of the change.&# 160; [Aside: We also need to figure out how to deal with messages that have already been sent to the failed node.]</p> <p>The same goes for the routers themselves.&# 160; Only one router is actively forwarding messages while the others are idling, listening to state changes from the active router and heartbeats from the nodes.&# 160; When the active router misses X heartbeats, the remaining routers elect a new leader.&# 160; (A super-easy formula for leader election is to use the one with highest <em>fixed</em> value, e.g. IP address, or # of CPUs, or # of GB of RAM, etc.)</p> <p>Once the new leader is elected, it takes over and starts processing messages and forwards them to the appropriate node.&# 160; When the failed router comes back online, it sees that another router is actively processing (probably through the heartbeat message from the newly elected primary router) and assumes its place in idle mode.&# 160; [Aside: How would it get the list of routes from the primary router?&# 160; Or is that information truly necessary?]</p> <p>When either a new or failed node comes online, it should publish a message to all routers as if it was a brand-new node because it probably doesn’t have any in-memory state any more.&# 160; Or even if it did, the routers may have redirected messages elsewhere.</p> <p><strong>Conclusion</strong></p> <p>Message-based systems can be exponentially more complex to orchestrate than simple request/reply models, but it’s because they’re exponentially more scalable.&# 160; Fortunately this complexity resides outside the domain and is in separate, reusable, infrastructure components so that you can grow into this solution as necessary.</p> <p>As a final thought, I would recommend that you have a set of routers for each bounded context that needs concurrency.</p>
