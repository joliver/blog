<p>Greg's <a href="http://codebetter.com/blogs/gregyoung/archive/2008/05/20/devteach-talk.aspx">devTeach talk</a> has various slides in which he presents a simple overview of persistence for event storage. In the video [at 39:30] he talks about how he doesn't use a relational database for storing his events, but he says that you could.</p> <p>Because I have only superficially glanced at solutions like <a href="http://en.wikipedia.org/wiki/BigTable">BigTable</a>, <a href="http://hypertable.org/">HyperTable</a>, and <a href="http://couchdb.apache.org/">CouchDB</a>, I'm still using a relational database on the back end. Here is a proposed schema for storing the state transitions for any and all events of a bounded context in a SQL Server relational database:</p> <p><font face="Courier New">CREATE TABLE [dbo].[Aggregates]      <br />(       <br />       [aggregate_id] [uniqueidentifier] NOT NULL,       <br />       [aggregate_type] [nvarchar](256) NOT NULL,       <br />       [snapshot_event_seq] [int] NOT NULL,       <br />       CONSTRAINT [PK_Aggregates] PRIMARY KEY CLUSTERED       <br />              ([aggregate_id] ASC)       <br />) </font></p> <p><font face="Courier New">CREATE TABLE [dbo].[Events]      <br />(       <br />       [aggregate_id] [uniqueidentifier] NOT NULL,       <br />       [event_seq] [int] NOT NULL,       <br />       [msg_type] [nvarchar](256) NOT NULL,       <br />       [msg_ver] [smallint] NOT NULL,       <br />       [msg_data] [varbinary](max) NOT NULL,       <br />       CONSTRAINT [PK_Events] PRIMARY KEY CLUSTERED       <br />              ([aggregate_id] ASC, [event_seq] ASC)       <br />)</font></p> <p><font face="Courier New" size="1"></font></p> <p></p> <p>Note that the data for the event messages is stored as a blob--much like serializing a DTO across the wire, but here message is serialized to a database.</p>
